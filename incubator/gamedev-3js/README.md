# Wish Garden

A small Three.js game that Codex can change while it runs in the Codex browser.
It is deliberately a game first. It has no embedded agent or Pi runtime.

## Question

What becomes easier when Codex can play a web game, change its source and
assets, and see those changes in the same workspace?

## Run

```bash
bun install
bun run dev
```

Open `http://localhost:3000`. Walk with WASD or the arrow keys. Click the
island to plant generated wish seeds.

The page polls `public/world.json`. These commands change the running world
without a refresh:

```bash
bun run world preset moonrise
bun run world preset ember
bun run world motes 20
bun run world lanterns 8
bun run world name The Place Between Ideas
```

World objects have stable IDs. Click one in the game to inspect it, then change
it from the terminal:

```bash
bun run world list
bun run world inspect moonkeeper
bun run world move moonkeeper 3 -2
bun run world scale moonkeeper 1.25
bun run world recolor moonkeeper '#b9ddff'
bun run world add wish-seed second-wish 1.5 2
```

`public/wish-seed.png` was generated by Codex with the built-in image generator,
then converted from a flat chroma-key background to an alpha PNG. The same
workflow produced `public/moon-tree.png` and the moon-moth creature in
`public/moon-moth.png`.

Import another PNG, JPEG, or WebP into the in-game placement catalog:

```bash
bun run entity import path/to/moon-moth.png
```

## Current seam

Codex owns repo edits, asset creation, commands, and browser feedback. The game
owns movement, collection, planting, and deterministic world state. That is
enough to expose useful friction before adding MCP tools or in-game agency.
